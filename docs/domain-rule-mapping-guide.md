# 领域规则映射指南

这份指南说明如何把一段自然语言规则映射到 Discrete Runtime。它只描述通用方法，
不要求项目采用某一种内容、资源、UI 或存档架构。

## 先写规则，再写类型

实现前先回答：

1. 哪些状态属于领域对象？
2. 当前处于什么阶段，阶段如何切换？
3. 谁可以在什么时机提出输入或行动？
4. 哪些变化只是请求，哪些变化已经实际发生？
5. 实际变化会触发哪些被动反应？
6. 什么条件结束当前阶段，什么条件终止整个流程？
7. 哪些工作需要等待外部输入、动画或其他异步结果？

Runtime 不替项目回答这些问题，它只为答案提供确定性的执行边界。

## 与 Runtime 概念对应

| 规则中的含义 | Runtime 概念 |
|---|---|
| 当前流程位置 | Phase |
| 请求阶段执行语义工作 | PhaseRequest |
| 请求修改领域状态 | Effect |
| 已经发生的状态变化 | Fact |
| 命令式工作或异步边界 | Operation |
| 正常完成当前阶段 | Result |
| 中断、终止或重定向 | Signal |
| 观察 Entry 并提出派生工作 | Handler |
| 修改状态并报告结果 | Applier |
| 执行 Operation | Processor |
| 稳定后产生阶段出口 | ResultEmitter |
| 解释出口并决定下一阶段 | Interpreter |

## 最小实现顺序

1. 用项目自己的对象保存领域状态，不把状态塞进 Runtime。
2. 画出最小 Phase 图，只保留完成闭环所需的阶段。
3. 将状态变化写成 `Effect → Applier → Fact`。
4. 将被动反应写成对 Fact 的观察，不在 Handler 中直接修改状态。
5. 将输入、动画等待或外部命令建模为 Operation。
6. 用 Result 表达正常阶段出口，用 Signal 表达中断或终止。
7. 先运行无 UI 场景，记录语义轨迹和最终状态。
8. 最后连接正式 UI、资源和表现层。

## Signal 的选择

- 真正需要截断后续工作时使用 `IMMEDIATE`。
- 出口已经确定，但现有逻辑仍要完整结算时使用 `AFTER_SETTLEMENT`。

结算模式不要求继续无意义的等待或演出。领域 Applier 与 Processor 可以检查
`workspace.is_settling_exit()`，让已经不适用的工作无阻塞完成。

## 确定性检查

- 身份不能依赖数组位置或对象地址。
- 同一优先级下的顺序应能由 Entry `seq` 解释。
- Handler、Applier、Processor 和 Emitter 的注册顺序必须显式构造。
- 随机选择应由领域层持有可复现的种子或随机源。
- 测试比较语义事件和状态，不依赖帧时间或控制台排版。

## 何时不应修改 Runtime

以下需求通常属于项目层：

- 行动候选、输入提交和修订号；
- 内容定义、实例、资源包或目录；
- UI 投影、隐私视图、动画协议；
- 存档、联网、随机数和回放格式；
- 某一种玩法专用的阶段、Effect 或 Fact。

只有当现有扩展点无法表达某项调度语义，并且同一缺口在多个不同领域中重复出现时，
才值得提出 Runtime 修改。修改提案应包含最小复现、期望顺序、现有轨迹和回归测试。
