# 跨玩法验证报告

## 目的

本报告记录 `v0.1.0` 公开前的一轮外部使用验证。验证目标不是证明案例本身已经达到
发布质量，而是检查陌生使用者能否在不修改 Runtime 的情况下表达不同领域和调度模型。

## 验证范围

同一份 `addons/discrete_runtime/` 源码被用于四类独立案例：

| 案例 | 主要压力 |
|---|---|
| 经典回合制角色扮演战斗 | 阶段轮转、行动选择、状态与被动反应 |
| CTB 时间轴战斗 | 以不同调度方式复用领域内容和结算结构 |
| 阵型战斗 | 位置、目标、长因果链和终止结算 |
| 颜色匹配卡牌 | 隐藏信息、嵌套选择和非战斗结束条件 |

另有一套卡牌战斗白盒曾用于早期集成验证，但它仍在继续完善，不作为 Runtime 首发的
完整性前提。

## 结果

反馈项目中的 Runtime `.gd` 源码与核心候选逐文件一致，没有为了案例增加领域特例。

验证时执行的测试为：

| 范围 | 测试 | 断言 |
|---|---:|---:|
| Runtime | 21 | 94 |
| 经典回合制 | 23 | 157 |
| CTB | 9 | 45 |
| 阵型战斗 | 15 | 73 |
| 颜色匹配卡牌 | 20 | 128 |
| 合计 | 88 | 497 |

全部测试通过。边界检查没有发现 Runtime 对案例类型、场景或资源的反向引用。

## 能够支持的结论

- Runtime 可以在不持有玩法状态的情况下承载多种离散流程。
- Effect/Fact、Operation 和双模式 Signal 能表达案例所需的主要因果与终止语义。
- 更换上层阶段调度方式不必然要求修改核心。
- 隐藏信息、输入协议、内容目录和 UI 投影可以保留在项目层。
- 当前没有来自这些案例的证据要求继续扩张 Runtime API。

## 不能支持的结论

- 这些结果不代表 Runtime 已覆盖所有游戏类型。
- 案例采用的 Offer、Commit、ContentPack、Projector 等结构不是 Runtime 强制架构。
- 测试通过不代表案例内容、UI 或发行资源已经达到独立发布质量。
- 当前只验证了 Godot 4.6.3 的 GDScript 参考实现，其他语言实现仍需各自验证。

## 后续使用

新增 Runtime 能力应优先由最小跨领域证据驱动。单个项目的便利性需求，应先尝试通过
领域 State、Phase、Entry、Handler、Applier、Processor、Emitter 或 Interpreter 解决。
